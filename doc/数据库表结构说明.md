# 数据库表结构说明

本文档根据当前项目中的 SQL 初始化脚本与 SQLAlchemy 模型整理，主要覆盖当前数据库中已生成/使用的业务表。

参考文件：

- `Backend/database_schema.sql`
- `Backend/sql/create_step_memory_tables.sql`
- `Backend/app/models/*.py`

## 一、整体用途概览

该数据库主要服务于 EvalFlow Pro / AgentSystem 后端，核心能力包括：

1. 用户、项目、文件资料管理。
2. 14 步智能体工作流的步骤输出、历史版本、异步任务记录。
3. Prompt、知识库、大模型配置的管理端维护。
4. LLM 调用统计、审计日志、对话消息记录。
5. 项目级短期/长期记忆与步骤运行历史保存。

## 二、表关系概览

主要关系如下：

- `users` 是用户主表。
- `projects` 通过 `user_id` 归属于 `users`。
- `files` 通过 `project_id`、`user_id` 归属于项目和用户。
- `step_outputs` 通过 `project_id` 保存项目每个步骤的最终输出。
- `step_histories` 通过 `step_output_id` 保存某个步骤输出的历史版本。
- `task_jobs` 通过 `project_id` 保存异步任务执行状态。
- `llm_calls` 通过 `user_id`、`project_id` 记录模型调用。
- `audit_logs` 通过 `actor_user_id` 记录用户操作审计。
- `conversations` 通过 `project_id`、`step_code` 记录项目步骤相关对话。
- `conversation_messages` 通过 `conversation_id` 记录对话消息。
- `project_memory_sessions` 是项目记忆会话。
- `project_memory_entries` 通过 `session_id` 归属于记忆会话。
- `project_step_runs` 记录每一次步骤工作流调用。

## 三、数据表详细说明

### 1. `users` 用户表

用途：保存系统登录账号、角色与账号状态，是权限、项目归属和审计日志的基础表。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 用户主键 ID，通常为 UUID 字符串。 |
| `username` | `VARCHAR(128)` | 非空，唯一 | 登录用户名。 |
| `password_hash` | `VARCHAR(255)` | 非空 | 密码哈希值，不保存明文密码。 |
| `role` | `VARCHAR(32)` | 非空，默认 `user` | 用户角色，例如 `user`、`admin`。 |
| `status` | `VARCHAR(32)` | 非空，默认 `active` | 用户状态，例如 `active`、`disabled`。 |

### 2. `projects` 项目表

用途：保存用户创建的项目，一个用户可以拥有多个项目。项目是文件、步骤输出、任务、对话和记忆的业务容器。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 项目主键 ID。 |
| `user_id` | `VARCHAR(36)` | 非空，索引，外键到 `users.id` | 项目所属用户 ID。 |
| `name` | `VARCHAR(255)` | 非空 | 项目名称。 |
| `description` | `TEXT` | 可空 | 项目描述。 |
| `status` | `VARCHAR(32)` | 非空，默认 `active` | 项目状态，例如 `active`、`archived`。 |

### 3. `files` 文件表

用途：保存项目上传资料的元信息，包括文件名、类型、存储位置、解析状态和元数据。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 文件主键 ID。 |
| `project_id` | `VARCHAR(36)` | 非空，索引，外键到 `projects.id` | 文件所属项目 ID。 |
| `user_id` | `VARCHAR(36)` | 非空，索引，外键到 `users.id` | 上传/所属用户 ID。 |
| `project_name` | `VARCHAR(255)` | 可空，索引 | SQLAlchemy 模型中存在，用于冗余保存项目名称，便于按项目名查询或展示。 |
| `file_name` | `VARCHAR(255)` | 非空 | 原始文件名称。 |
| `file_type` | `VARCHAR(64)` | 非空 | 文件类型或后缀，例如 `pdf`、`docx`、`xlsx`。 |
| `storage_key` | `VARCHAR(512)` | 非空 | 文件在本地或对象存储中的路径/Key。 |
| `parse_status` | `VARCHAR(32)` | 非空，默认 `pending` | 文件解析状态，例如 `pending`、`parsed`、`failed`。 |
| `source_type` | `VARCHAR(32)` | 可空 | 资料来源类型，例如 `media`、`documents`。 |
| `file_size` | `INTEGER` | 可空 | 文件大小，单位通常为字节。 |
| `metadata_json` | `TEXT` | 可空 | 文件元数据 JSON 字符串。 |

> 注意：`project_name` 在 `Backend/database_schema.sql` 中未出现，但在当前 `Backend/app/models/file.py` 模型中存在。如果数据库由模型自动建表/迁移生成，该字段会出现在数据库中。

### 4. `step_outputs` 步骤最终输出表

用途：保存每个项目、每个步骤的最终成品输出。适用于 14 步工作流中每一步的结构化结果或文本结果。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 步骤输出主键 ID。 |
| `project_id` | `VARCHAR(36)` | 非空，索引，外键到 `projects.id` | 所属项目 ID。 |
| `step_code` | `VARCHAR(32)` | 非空，索引 | 步骤编码，例如 `step1`、`step2`。 |
| `title` | `VARCHAR(255)` | 非空 | 输出标题。 |
| `content_json` | `TEXT` | 可空 | 结构化输出 JSON 字符串。 |
| `content_text` | `TEXT` | 可空 | 纯文本输出内容。 |
| `version` | `INTEGER` | 非空，默认 `1` | 输出版本号。 |
| `is_final` | `BOOLEAN` | 非空，默认 `FALSE` | 是否为最终版本。 |

### 5. `step_histories` 步骤历史表

用途：保存步骤输出的历史版本、多轮修改结果或多模型对比结果。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 历史记录主键 ID。 |
| `step_output_id` | `VARCHAR(36)` | 非空，索引，外键到 `step_outputs.id` | 关联的步骤最终输出 ID。 |
| `model_name` | `VARCHAR(128)` | 可空 | 生成该历史版本的模型名称。 |
| `prompt_version_id` | `VARCHAR(36)` | 可空，外键到 `prompt_versions.id` | 关联的 Prompt 版本 ID。 |
| `content_json` | `TEXT` | 可空 | 历史结构化内容 JSON。 |
| `content_text` | `TEXT` | 可空 | 历史文本内容。 |

### 6. `task_jobs` 异步任务表

用途：保存异步任务的执行状态，例如步骤生成、文件解析、导出等任务。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 任务主键 ID。 |
| `project_id` | `VARCHAR(36)` | 非空，索引，外键到 `projects.id` | 所属项目 ID。 |
| `step_code` | `VARCHAR(32)` | 非空，索引 | 任务对应的步骤编码。 |
| `task_type` | `VARCHAR(64)` | 非空 | 任务类型，例如 `generate`、`export`、`parse`。 |
| `status` | `VARCHAR(32)` | 非空，默认 `pending` | 任务状态，例如 `pending`、`running`、`succeeded`、`failed`、`canceled`。 |
| `progress` | `INTEGER` | 非空，默认 `0` | 任务进度百分比。 |
| `error_message` | `TEXT` | 可空 | 任务失败时的错误信息。 |

### 7. `prompt_versions` Prompt 版本表

用途：管理端维护不同步骤使用的 Prompt 内容和版本，支持按步骤启用某个 Prompt 版本。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | Prompt 版本主键 ID。 |
| `step_code` | `VARCHAR(32)` | 非空，索引 | 适用步骤编码。 |
| `version` | `INTEGER` | 非空，默认 `1` | Prompt 版本号。 |
| `title` | `VARCHAR(255)` | 非空 | Prompt 标题。 |
| `content` | `TEXT` | 非空 | Prompt 完整内容。 |
| `is_active` | `BOOLEAN` | 非空，默认 `FALSE` | 是否为当前启用版本。 |

### 8. `kb_versions` 知识库版本表

用途：管理端维护每个步骤可使用的知识库文件或知识库引用，支持多版本管理和启用状态控制。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 知识库版本主键 ID。 |
| `step_code` | `VARCHAR(32)` | 非空，索引 | 适用步骤编码。 |
| `version` | `INTEGER` | 非空，默认 `1` | 知识库版本号。 |
| `name` | `VARCHAR(255)` | 非空 | 知识库名称。 |
| `storage_ref` | `TEXT` | 非空 | 知识库存储引用、路径或对象存储 Key。 |
| `is_active` | `BOOLEAN` | 非空，默认 `FALSE` | 是否为当前启用版本。 |

### 9. `llm_calls` 大模型调用记录表

用途：记录每次大模型调用的用户、项目、步骤、模型、Token 消耗、耗时、请求和响应内容，用于用量统计、成本分析和问题追踪。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 调用记录主键 ID。 |
| `user_id` | `VARCHAR(36)` | 非空，索引，外键到 `users.id` | 调用用户 ID。 |
| `project_id` | `VARCHAR(36)` | 非空，索引，外键到 `projects.id` | 所属项目 ID。 |
| `step_code` | `VARCHAR(32)` | 非空，索引 | 调用所属步骤编码。 |
| `model_name` | `VARCHAR(128)` | 非空 | 使用的模型名称。 |
| `prompt_tokens` | `INTEGER` | 非空，默认 `0` | 输入 Prompt 消耗 Token 数。 |
| `completion_tokens` | `INTEGER` | 非空，默认 `0` | 模型输出消耗 Token 数。 |
| `total_tokens` | `INTEGER` | 非空，默认 `0` | 总 Token 数。 |
| `latency_ms` | `INTEGER` | 非空，默认 `0` | 调用耗时，单位毫秒。 |
| `request_payload` | `TEXT` | 可空 | 请求内容 JSON 字符串。 |
| `response_payload` | `TEXT` | 可空 | 响应内容 JSON 字符串。 |

### 10. `audit_logs` 审计日志表

用途：记录系统关键操作变更，例如创建、更新、删除、发布配置等，便于追踪管理端操作。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 审计日志主键 ID。 |
| `actor_user_id` | `VARCHAR(36)` | 非空，索引，外键到 `users.id` | 执行操作的用户 ID。 |
| `action` | `VARCHAR(64)` | 非空 | 操作动作，例如 `create`、`update`、`delete`、`publish`。 |
| `target_type` | `VARCHAR(64)` | 非空 | 操作对象类型，例如 `project`、`prompt`、`kb`。 |
| `target_id` | `VARCHAR(36)` | 非空，索引 | 操作对象 ID。 |
| `before_data` | `TEXT` | 可空 | 变更前数据 JSON 字符串。 |
| `after_data` | `TEXT` | 可空 | 变更后数据 JSON 字符串。 |

### 11. `model_registry` 模型配置表

用途：保存管理端配置的大模型接入信息，包括模型名称、模型 ID、Base URL、API Key、启用状态和是否支持多模态。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 模型配置主键 ID。 |
| `name` | `VARCHAR(128)` | 非空 | 模型显示名称。 |
| `api_key` | `TEXT` | 可空 | 模型 API Key。建议生产环境加密存储。 |
| `base_url` | `TEXT` | 可空 | 模型服务基础地址。 |
| `model_id` | `VARCHAR(128)` | 非空 | 实际调用时使用的模型 ID。 |
| `enabled` | `BOOLEAN` | 非空，默认 `TRUE` | 是否启用该模型配置。 |
| `supports_vision` | `BOOLEAN` | 非空，默认 `FALSE` | 是否支持图片、PDF 等多模态能力。 |

### 12. `conversations` 对话会话表

用途：保存项目中某个步骤相关的 AI 对话会话，用于连续追问、修改建议、上下文讨论等场景。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `VARCHAR(36)` | 主键 | 对话会话主键 ID。 |
| `project_id` | `VARCHAR(36)` | 非空，索引 | 所属项目 ID。 |

| `title` | `VARCHAR(255)` | 非空，默认 `AI 对话` | 对话标题。 |

| `created_at` | `TIMESTAMPTZ` | 非空，默认当前时间 | 创建时间。 |

## 四、当前表的业务分组

### 用户与项目基础数据

- `users`
- `projects`
- `files`

这些表支撑登录、项目创建、项目资料上传和资料解析。

### 步骤工作流数据

- `step_outputs`
- `step_histories`
- `task_jobs`
- `project_step_runs`

这些表用于保存每一步生成结果、历史修改、异步执行状态和实际运行日志。

### 管理端配置数据

- `prompt_versions`
- `kb_versions`
- `model_registry`

这些表用于维护 Prompt、知识库和模型配置。

### 对话与记忆数据

- `conversations`
- `conversation_messages`
- `project_memory_sessions`
- `project_memory_entries`

这些表用于保存用户与 AI 的持续对话，以及项目/步骤级上下文记忆。

### 统计与审计数据

- `llm_calls`
- `audit_logs`

这些表用于模型调用统计、成本分析、问题追踪和关键操作审计。

## 五、补充建议

1. 建议为所有带 `created_at`、`updated_at` 语义的业务表统一时间字段，当前部分核心表如 `users`、`projects`、`files`、`step_outputs` 暂未在 SQL 脚本中声明时间字段。
2. `files.project_name` 当前存在于模型中，但不在 `Backend/database_schema.sql` 中，建议统一 SQL 初始化脚本和 ORM 模型，避免环境差异。
3. `project_memory_sessions.project_id`、`project_memory_sessions.user_id`、`project_step_runs.project_id` 等字段当前 SQL 中未声明外键，如需更强数据一致性，可补充外键约束。
4. `model_registry.api_key` 涉及敏感信息，建议生产环境加密存储，避免明文落库。
5. 如果后续使用 Alembic 迁移，建议以迁移脚本作为唯一结构来源，避免 SQL 文件和 ORM 模型不一致。
