# 技术栈

> 产品名：云海睿评（网页版，云服务器运行）
> 
> 目标：用“大模型 + 提示词 + 知识库”固化绩效评价报告撰写流程，提高效率与质量；支持多人并发、数据隔离。

## 1. 你当前的技术栈是否够用？（结论）
你当前给出的组合：
- 前端：Vue
- 后端：FastAPI
- 智能体编排：LangChain + LangGraph
- 监控：LangSmith
- 数据库：PostgreSQL

**整体是够用的**，可以覆盖：Web 产品交互、流程编排、LLM 调用、多模型对比、Prompt/知识库管理、用户/权限/审计、项目与产物存储、token 统计。

但从“一个人开发、上线可用、可运维”的角度，还需要补齐一些关键配套能力：
- 鉴权/权限与多租户隔离（组织/用户/项目维度）
- 文件上传、解析、向量化检索（RAG）
- 异步任务队列（OCR/解析/长文本生成/多模型对比都需要）
- 对象存储（文档、导出成果、项目资料）
- 文档导出（Word/PDF 排版与模板）
- 可观测性（业务审计日志 + 性能指标 + 报错追踪）
- 部署与运维（容器化、反向代理、HTTPS、CI/CD、备份）

## 2. 技术栈统计（推荐补齐项）

### 2.1 前端（Vue 侧）
- 框架：Vue 3 + TypeScript
- 构建：Vite
- UI 组件：
  - 推荐：Naive UI / Element Plus（二选一，保证后台管理与表格能力强）
- 状态管理：Pinia
- 路由：Vue Router
- 数据请求：Axios
- 富文本/结构化编辑：
  - 指标体系与评分标准更像“结构化表格编辑”，建议：
    - 表格：AG Grid / vxe-table（二选一，支持拖拽排序、合并单元格、复杂编辑）
    - 富文本：Tiptap（用于报告拼装后的段落编辑）
- 文件上传：分片上传（可选），进度条、断点续传（后期）
- 权限控制：前端路由守卫 + 权限指令（配合后端 RBAC）

### 2.2 后端（FastAPI 侧）
- Web 框架：FastAPI
- ASGI Server：Uvicorn / Gunicorn(Uvicorn workers)
- 数据访问：SQLAlchemy 2.0 + Alembic（迁移）
- 校验：Pydantic v2
- 鉴权：
  - JWT（access/refresh）
  - 密码哈希：passlib(bcrypt)
  - RBAC：角色（管理员/普通用户）+ 资源权限
- 异步任务队列（强烈建议）：
  - 方案 A：Celery + Redis
  - 方案 B：RQ + Redis（更轻）
  - 方案 C：Dramatiq + Redis
  - 用途：OCR、文档解析、embedding、长文本生成、多模型并行对比、导出 PDF/Word
- 实时进度与通知：WebSocket / SSE（任务进度条、生成流式输出）

### 2.3 LLM/智能体与 RAG
- 编排：LangGraph（主）+ LangChain（工具与集成）
- 模型接入：
  - OpenAI/兼容 OpenAI API 的平台（你文档里提到：管理端可配置 API Key、Base URL、模型ID）
  - 需要统一“模型路由层”（model registry）
- 多模型对比：并行调用 + 结果归档 + 人工选择“最终版本”
- RAG（知识库）：
  - 文档切分：langchain text splitters
  - Embedding：按你选择的模型供应商
  - 向量库：
    - 轻量推荐：pgvector（直接用 PostgreSQL）
    - 或：Qdrant（独立向量库，易用）
  - 检索：Hybrid（向量 + 关键词，后期）
- OCR/解析：
  - PDF 文本：pypdf/pdfplumber
  - 扫描件/图片：PaddleOCR（中文效果好）或 Tesseract（备选）

### 2.4 数据库与存储
- 主库：PostgreSQL
- 向量：pgvector（推荐一起用）
- 对象存储（强烈建议）：
  - MinIO（自建 S3 兼容）/ 阿里云 OSS / 腾讯云 COS
  - 存：用户上传原始资料、过程产物、导出 Word/PDF、日志附件
- 缓存/队列：Redis（缓存、任务队列、限流）

### 2.5 观测、审计、监控
- LLM 观测：LangSmith（已选）
- 日志：structlog / loguru + JSON log
- 指标监控：Prometheus + Grafana（可选，但建议至少准备接口）
- 错误追踪：Sentry（强烈建议，省时间）
- 业务审计：
  - 关键动作：登录、模型配置变更、Prompt/知识库变更、导出、删除、下载
  - 需要“谁在何时改了什么”的变更日志（文档明确要求）

### 2.6 部署与工程化
- 容器：Docker + docker-compose（单人最友好）
- 反向代理与 HTTPS：Nginx / Caddy（二选一）
- CI/CD：GitHub Actions / GitLab CI（后期）
- 数据备份：PostgreSQL 定时备份 + 对象存储生命周期策略
- 安全：
  - HTTPS、密码策略、API 限流、上传文件安全扫描（后期可加）

### 2.7 文档导出与模板（本产品核心能力之一）
文档大量要求“一键导出 Word/PDF”“经典排版/自定义排版”。建议：
- Word：python-docx（适合结构化生成）
- PDF：
  - 方案 A：先生成 Word 再转 PDF（libreoffice headless）
  - 方案 B：WeasyPrint（HTML->PDF，排版可控）
- 模板：
  - docx 模板占位符（Jinja2 + docx-template/docxtpl）
  - 指标体系表格输出（Word 表格/HTML 表格）

## 3. 你还没明确但建议纳入的“产品级能力”
- 多租户隔离：至少做到“用户只能看到自己项目与产物”；管理端可跨用户查看。
- 90 天资料保留：对象存储生命周期 + 数据库软删除/过期任务。
- Token/费用统计：按请求记录 model、prompt_tokens、completion_tokens、total_tokens、耗时、用户、项目、步骤。
- Prompt/知识库可视化配置：版本化、发布/回滚、修改日志。
- 任务与历史记录：主界面只保留最终版本，历史在“历史记录”独立页面（文档明确）。
- 模型温度/Top-p 三档：发散/普通/严谨（离散参数映射）。

# 附：从 Word（提取文本）总结出的产品要开发什么

## A. 产品定位与整体架构
- 产品名：云海睿评
- 形态：网页版，云服务器部署
- 用户体系：
  - 管理端：账号管理、模型与 Prompt/知识库配置、审计、查看 token、使用时长、下载全量资料
  - 用户端：登录、创建项目、上传资料、按步骤生成内容、历史记录、导出、查看个人消耗记录
- 并发与隔离：多人同时使用互不影响，用户端不可互相查看记录
- 核心：用 LLM+Prompt+知识库，固化“绩效评价报告撰写”的工作流

## B. 用户端核心工作流模块（按步骤）
下面 1-14 步是“部门评价/财政评价报告撰写系统”的主流程。每步普遍包含：
- 上传/输入
- 调用模型生成
- 人机交互（人工直接改 or 与模型多轮对话改）
- 多模型对比（可选）
- 选择最终版本 -> 保存到“成品”
- 导出 Word/PDF（经典/自定义排版）

### 1）生成资料清单
- 输入：基础项目资料（word/excel/pdf/图片）
- 输出：项目资料清单（可对话优化、可手改），导出“项目资料清单”

### 2）生成有效项目资料（核心内容）
- 输入：按清单收集的详细资料
- 特性：图片/PDF 与 Word 分开上传；考虑部分模型不识别图片/PDF；需要人工校验
- 输出：剔除无效后的核心内容，按类别归类；带“索引/溯源”（能对回原文）

### 3）生成指标体系
- 基于第二步核心内容生成 1/2/3 级（特殊可 4 级）指标体系
- 支持导入自有指标体系、从模板库导入并一键优化
- 支持逐个二级指标拆解生成三级指标及解释（可对话/对比）

### 4）生成分值
- 对指标体系赋分：模型自动赋分或人工赋分
- 强制校验：层级合计=100、父子分值一致、预警（产出+效益 < 60 等规则）

### 5）生成评分标准
- 针对每个三级指标生成评分标准（逐个确认、可对话/对比）

### 6）生成完整版绩效评价指标体系
- 可视化表格：拖拽排序（一级/二级/三级不同约束）
- 输出表格列：一级/二级/三级、分值、指标解释、评分标准
- 可选：基于指标体系生成调查问卷（里克特 5 级，默认不含填空，可人工加）

### 7）生成指标体系分析内容（并形成得分表）
- 输入：项目核心内容 + 现场评价数据
- 机制：评分标准每个要点都有输入框，操作者写“得分/扣分原因”，模型生成分析文本
- 输出：
  - 各一级指标分析（决策/过程/产出/效益）
  - 二级小计、自动汇总
  - 最终“指标体系得分表”（含扣分原因）

### 8）生成经验做法
- 输入：核心内容 + 现场评价 + 工作总结
- 输出：经验做法段落

### 9）生成问题及原因分析
- 从第七步扣分点“一键提取”并由操作者选择是否纳入问题
- 支持手动增补问题
- 输出：问题+原因分析，支持语气风格（中立/尖锐/委婉）

### 10）生成建议
- 针对每个问题生成建议，支持语气风格
- 系统做逻辑校对：指标体系/分析/问题/建议之间一致性

### 11）生成综合评价分析情况及评价结论
- 自动汇总评分并判定等级（优秀/良好/合格/不合格等可配置）
- 结构化综合分析：整体概况、亮点、问题脉络、结论与总览

### 12）生成基本情况
- 生成模块化“背景/内容/组织管理/资金投入/绩效目标”等，可增模块

### 13）绩效评价工作开展情况
- 生成“目的、对象、范围、原则、指标、方法、标准、过程”等

### 14）生成评价报告（拼装与格式化）
- 一键抽取第六步至第十三步成果，按固定顺序或自定义顺序组装
- 支持全文预览/修改/段落调整
- 按样本集/规范（如 396 号文要求）做格式校对并导出最终 Word/PDF

## C. 管理端模块（贯穿每一步）
- 账号管理：新增/分发账号密码、查看登录时间与使用时长
- 成本统计：查看每个报告消耗 tokens
- 模型配置（可视化）：平台开关、API Key、Base URL、模型增删，用户端实时同步
- Prompt/知识库配置（可视化）：
  - 每一步可配置
  - 部分步骤可细化到“某组二级指标”的专用 Prompt/知识库
  - 需要修改日志（谁在何时改了什么）
  - 客户端不可查看详细 Prompt/知识库（保护知识产权）
- 数据：管理端/用户端数据融通，但权限隔离

# 开发所需文档（你一个人开发，建议这样写）

## 1) 前端开发文档（建议大纲）
1. 页面结构与路由
   - 登录/注册
   - 用户端：项目列表/项目详情（14 步导航）/历史记录/消费记录/个人中心
   - 管理端：用户管理/模型管理/Prompt&知识库管理/审计日志/用量统计/全量下载
2. 关键交互规范
   - “生成区（多模型对比）” + “成品区（最终版本）” + “历史记录页”三分结构
   - 多模型并行结果的对比 UI（分栏/Tab/差异高亮）
   - 表格拖拽排序（指标体系）
   - 长任务进度（排队/运行/完成/失败）
   - 流式输出（打字机效果）与中途停止
3. 上传与资料管理
   - 文件类型限制、大小限制、分组上传（图片/PDF vs Word）
   - 上传进度、失败重试
4. 导出与排版设置
   - 经典排版 vs 自定义（字体/字号/行距/段落）
   - 预览与导出入口规范
5. 权限与多租户
   - 用户端不可见他人项目
   - 管理端跨用户可见
6. 前端错误处理与埋点
   - 统一错误提示
   - 关键操作埋点（生成、导出、保存、对比选择）

## 2) 后端开发文档（建议大纲）
1. 领域模型与数据表（最低集合）
   - users, roles
   - projects
   - files (metadata)
   - steps_outputs（每一步的“成品版本”）
   - step_histories（历史版本/多模型对比结果）
   - prompt_versions / kb_versions / change_logs
   - llm_calls（token、耗时、模型、步骤、用户、项目）
2. API 设计
   - Auth：注册/登录/刷新 token
   - Project：CRUD
   - File：上传/下载/列表
   - Step：触发生成（异步）/查询任务/保存成品/获取历史
   - Export：导出 word/pdf
   - Admin：模型配置、Prompt/KB 配置、审计、用量统计
3. 异步任务与幂等
   - 任务状态机：queued/running/succeeded/failed/canceled
   - 重试策略与超时
4. 文档解析与索引
   - 文本抽取/OCR
   - 溯源索引（chunk -> 原文位置/页码）
5. LLM 编排规范
   - 每一步一个 graph 节点/子图
   - 参数：温度/Top-p 三档映射
   - 多模型并行：fan-out/fan-in
6. 权限、安全与审计
   - RBAC
   - 审计日志不可篡改（至少追加写）
7. 数据保留与清理
   - 90 天游程：定时任务清理对象存储与数据库记录
