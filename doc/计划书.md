# EvalFlow Pro 从 0 到开发完成：计划书（单人可落地）

> 产品名：EvalFlow Pro
>
> 目标：将“绩效评价报告撰写”固化为可配置工作流（大模型 + Prompt + 知识库），支持多用户并发隔离、可视化管理端配置、全流程留痕、Word/PDF 导出。

---

## 0. 总体交付标准（Definition of Done）

### 用户端
- 登录后可创建项目、上传资料、按步骤生成内容
- 主操作界面仅保留“最终版本”，历史版本在“历史记录”页可回溯
- 至少完成 **MVP 闭环**（见 Phase 0）并能导出 Word（PDF 可后置）
- 用户仅能查看自己的项目、资料、历史、导出物
- 项目资料与产物自上传起最长保留 90 天（自动清理）

### 管理端
- 可视化配置：模型列表（Base URL / API Key / model_id / 开关）
- 可视化配置：每一步 Prompt、知识库（版本化、发布/回滚）
- 变更日志：谁在何时改了什么（不可被客户端查看 Prompt/知识库细节）
- 用量与审计：按用户/项目/步骤统计 tokens、耗时、登录与使用时长
- 可下载所有用户的报告与项目资料（权限受控）

### 工程质量
- 所有长耗时任务异步化（可查询进度、可取消、失败可重试）
- 关键链路可观测（LLM 调用链路、tokens、耗时、错误）
- Harness 回归可跑（LangSmith）

---

## 1. MVP 纵切建议（先做最短闭环）

强烈建议先把以下步骤打通形成可交付闭环：
- **Step2 生成有效项目资料（核心内容）**
- **Step3 生成指标体系**
- **Step4 生成分值（含校验）**
- **Step5 生成评分标准**
- **Step14 拼装与导出评价报告（MVP 版）**

为什么这样选：这是从“资料 → 指标体系 → 赋分/标准 → 可导出成稿”的最短路径，最能验证产品价值，且能为后续步骤复用数据结构。

MVP 先不做或弱化：
- 全量 14 步一次性做齐
- 复杂的“完全自定义排版”与高度可编辑 PDF（先 Word 稳定）
- 多模型对比做 2 个模型并行即可（后续再扩展）

## 2. Phase 0：需求冻结与范围定版（0.5–1 天）

### 关键决策（必须先定）
1. MVP 覆盖步骤：是否采用 Step2/3/4/5/14
2. 导出策略：先 Word 后 PDF（推荐）
3. 多模型对比：MVP 2 模型并行 + 人工选“最终版本”

### 交付物
- 《MVP 范围说明（1 页）》：做什么/不做什么
- 《里程碑与验收标准（本计划书即为基础）》

---

## 3. Phase 1：工程骨架与基础设施（2–4 天）

### 后端（FastAPI）
- FastAPI + PostgreSQL + Alembic
- JWT（access/refresh）+ RBAC（admin/user）
- 统一错误码与异常处理
- 结构化日志（JSON）与 request_id
- Redis 接入（后续队列/缓存/限流）

### 前端（Vue）
- Vue3 + TS + Vite
- UI 框架二选一：Naive UI / Element Plus
- 路由骨架：登录/注册、用户端壳、管理端壳
- 通用组件空壳：上传组件、任务进度组件、生成区/成品区/历史区布局

### 验收标准
- 本地可一键启动（推荐 docker-compose 起 pg/redis；对象存储 Phase3 再加）
- 注册/登录可用；admin 与 user 权限隔离生效

---

## 4. Phase 2：核心数据模型与 API 底座（2–5 天）

### 数据模型（最小集合）
- `users`, `roles`
- `projects`
- `files`（文件元数据 + object_key）
- `step_outputs`（每一步“最终版本”）
- `step_histories`（多轮/多模型历史版本）
- `llm_calls`（model、tokens、latency、user_id、project_id、step）
- `prompt_versions`, `kb_versions`, `change_logs`
- `audit_logs`（登录、导出、配置变更、下载等）

### API（最小集合）
- Auth：注册/登录/刷新 token
- Project：创建/列表/详情
- File：上传/列表/下载（先仅元数据+存储）
- Step：
  - 触发生成（异步任务 id）
  - 查询任务状态/进度
  - 保存“设为最终版本”
  - 查询历史版本

### 验收标准
- 能创建项目、上传文件、保存某一步“成品”和“历史”并可回读展示

---

## 5. Phase 3：文件上传 + 解析/OCR + RAG（5–10 天，最关键）

### 目标
把输入资料转为可用的“可检索文本”，并支持溯源（引用到原文）。

### 推荐实现顺序
1. 对象存储：MinIO（S3 兼容）
2. 文本抽取：
   - PDF 文本：pdfplumber/pypdf
   - Word：python-docx
   - Excel：openpyxl（按需）
3. OCR：
   - 图片/扫描 PDF：PaddleOCR（中文优先）
4. 切分与向量化：
   - 文本切分 + embedding
   - 向量库：优先 `pgvector`（少组件），或 Qdrant
5. 溯源：chunk 存储 file_id、page、offset、snippet

### 验收标准
- 上传一份混合资料后，能形成项目语料库并可检索返回“引用片段 + 来源”

---

## 6. Phase 4：异步任务队列 + LangGraph 编排 + 多模型对比（7–14 天）

### 目标
所有“生成/解析/导出”都异步化，可并行、可取消、可追踪。

### 必做
- 队列三选一：
  - RQ + Redis（更适合单人，推荐）
  - Celery + Redis（更重但成熟）
  - Dramatiq + Redis
- 任务状态机：queued/running/succeeded/failed/canceled
- LangGraph：每一步一个子图（检索→提示词组装→生成→结构化校验→落库）
- 多模型对比：fan-out 并行 → 写入 histories → 前端选择“设为最终版本”
- 进度与流式输出：SSE/WebSocket（至少支持进度查询与增量输出）

### 验收标准
- 任一步触发生成：可看到排队/运行/完成/失败；完成后可选择版本并保存为成品

---

## 7. Phase 5：完成 MVP 闭环（7–14 天）

### MVP 步骤实现要点
1. Step2 核心内容：
   - 分类输出（资金/预算/制度/实施等）
   - 必须包含引用与溯源
2. Step3 指标体系：
   - 输出结构化 JSON（一级/二级/三级）
   - 支持手改与模板导入（模板可后置）
3. Step4 赋分：
   - 自动赋分 + 人工修订
   - 分值校验（层级合计与预警规则）
4. Step5 评分标准：
   - 逐个三级指标生成，支持确认/修改
5. Step14 拼装导出：
   - 将 Step2/3/4/5 的成品按顺序组装
   - 导出 Word（先稳定）

### 验收标准
- 真实项目从上传 → Step2/3/4/5 → Step14 导出 Word 全链路走通
- 历史版本可回溯；最终版本在主界面清晰呈现

---

## 8. Phase 6：管理端能力上线（5–10 天）

### 必做模块
- 模型配置（可视化）：开关、API Key、Base URL、模型增删，用户端实时同步
- Prompt/知识库配置（可视化）：按步骤版本化、发布/回滚
- 变更日志：谁、何时、改了什么（客户端不可见 Prompt/KB 细节）
- 用量统计：按用户/项目/步骤汇总 tokens、耗时
- 全量下载：按用户/项目打包下载（zip）

### 验收标准
- 管理端改 Prompt 并发布后，用户端下一次生成使用新版本；变更日志可查

---

## 9. Phase 7：扩展到全 14 步（2–6 周，迭代交付）

建议按依赖顺序扩展：
1. Step6 指标体系表格化 + 问卷生成
2. Step7 指标分析 + 得分表（输入框驱动）
3. Step9/10 问题原因/建议 + 逻辑校对
4. Step11 综合评价结论（含等级判定与可视化）
5. Step12/13 基本情况/工作开展情况
6. Step1/8 资料清单/经验做法

每新增一步，都遵循同一套标准：异步任务 + 历史版本 + 成品版本 + 导出 + tokens 统计。

---

## 10. Phase 8：LangSmith Harness（贯穿建设，建议从 Phase 4 开始）

### Harness 的定位
- 用于 Prompt/知识库/模型路由变更后的回归评测
- 先覆盖关键步骤（Step2/3/5/11），逐步扩展

### 用例资产（建议结构）
- 样例输入：脱敏后的项目语料（或抽取文本）+ 关键字段
- 断言与指标：
  - schema 校验（结构化输出必须过）
  - 引用存在性（必须给出来源）
  - 规则校验（分值合计、层级一致）
  - 可选：LLM-as-judge（后期）

### 验收标准
- 每次发布 Prompt/KB 版本前：一键跑评测 → LangSmith 出报告 → 支持对比前后版本

---

## 11. 你现在就可以开始的“第一周路线图”（强执行版）

- Day 1：确定 MVP 范围 + 工程初始化（前后端）
- Day 2：鉴权 + RBAC + 项目 CRUD
- Day 3：文件上传（先落对象存储）+ files 表
- Day 4：step_outputs / step_histories 数据模型 + 基础 API
- Day 5：引入队列（RQ/Celery）+ 任务状态查询接口

第一周结束验收：
- 用户能登录、建项目、传文件、触发一个“空任务”并看到状态变化，为后续 LLM/解析铺路。

---

## 12. 风险清单（提前规避）
- OCR 与解析质量不稳定：先限定支持的文件类型与大小，逐步扩展
- Word/PDF 排版返工：先锁定 Word 模板策略（docxtpl），PDF 后置
- 14 步范围过大：必须坚持纵切 MVP
- Prompt/知识库频繁改动导致质量漂移：Harness 必须从早期就接入

---

## 13. 需要你最终确认的 3 个选项（用于落地排期与技术选型）
1) MVP 是否采用 Step2/3/4/5/14
2) 队列：RQ 还是 Celery
3) 向量库：pgvector(PostgreSQL) 还是 Qdrant
