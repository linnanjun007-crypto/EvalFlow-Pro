# 后端开发方案（FastAPI + LangGraph Agent）

> 项目：云海睿评 / EvalFlow Pro
>
> 目标：把 `Backend/agent/src/agent/user_graphs` 与 `Backend/agent/src/agent/admin_graphs` 封装为可由 FastAPI 调用的业务 API，并与前端联调打通。

---

## 1. 项目定位

本项目是一个面向绩效评价报告撰写的 Web 系统，核心流程是：

- 用户登录
- 创建项目
- 上传资料
- 通过步骤化工作流生成内容
- 支持多模型对比
- 支持人工修改与多轮对话优化
- 保存历史版本与最终成品
- 导出 Word / PDF
- 管理端维护 Prompt、知识库、模型配置与审计日志

后端采用 FastAPI 作为对外 API 层，LangGraph 作为智能体执行层，PostgreSQL 作为主数据库，Redis 作为任务与缓存组件。

---

## 2. 当前 `agent` 目录的判断

### 2.1 `Backend/agent/src/agent/graph.py`
它是一个统一路由入口，负责根据 `workflow_role` 路由到：

- `user_graphs`
- `admin_graphs`

它有用，但不是业务主流程，只是总入口与调度层。

### 2.2 `Backend/agent/src/agent/user_graphs`
该目录已经具备真实业务雏形，当前重点步骤包括：

- `step1`：资料清单生成
- `step2`：有效项目资料 / 核心内容生成

这两个步骤已经包含：

- 文件解析
- 多模型对比
- 人工 review
- 最终确认
- 导出字段

### 2.3 `Backend/agent/src/agent/admin_graphs`
当前已具备管理端雏形，主要是：

- Prompt 配置
- 知识库配置
- 配置草稿与最终确认

### 2.4 结论
这些 graph 可以直接被 FastAPI 包装成业务 API，前端通过 FastAPI 间接调用 agent，不直接接触 LangGraph。

---

## 3. 推荐总体架构

```text
前端 Vue
   ↓
FastAPI API 层
   ↓
Service 业务层
   ↓
LangGraph Agent 执行层
   ↓
PostgreSQL / Redis / MinIO / LLM 服务
```

### 各层职责

#### FastAPI
- 接收前端请求
- 鉴权
- 参数校验
- 返回统一 JSON
- 暴露项目 / 文件 / 步骤 / 导出 / 管理端接口

#### Service 层
- 组织 LangGraph 的 state/context
- 调用对应 step graph
- 保存输出与历史版本
- 创建任务记录
- 处理业务规则

#### LangGraph Agent
- 执行步骤逻辑
- 生成草稿
- 多模型对比
- review / approve 流程
- 输出结构化结果

#### 数据层
- 用户、项目、文件、任务、步骤输出、历史版本、Prompt 版本、知识库版本、审计日志

## 4. 后端要做哪些 API

下面按模块列出建议 API。

---

### 4.1 鉴权 API

#### `POST /api/auth/register`
注册用户。

#### `POST /api/auth/login`
用户登录，返回 access token / refresh token。

#### `POST /api/auth/refresh`
刷新 token。

#### `POST /api/auth/logout`
退出登录。

#### `GET /api/auth/me`
获取当前登录用户信息。

---

### 4.2 用户与权限 API

#### `GET /api/users/me`
获取当前用户资料。

#### `PATCH /api/users/me`
修改个人资料。

#### `GET /api/admin/users`
管理员查看用户列表。

#### `PATCH /api/admin/users/{user_id}/status`
管理员启用或禁用用户。

---

### 4.3 项目 API

#### `POST /api/projects`
创建项目。

#### `GET /api/projects`
项目列表。

#### `GET /api/projects/{project_id}`
项目详情。

#### `PATCH /api/projects/{project_id}`
修改项目名称、描述等。

#### `DELETE /api/projects/{project_id}`
删除项目。

---

### 4.4 文件 API

#### `POST /api/projects/{project_id}/files`
上传资料文件。

#### `GET /api/projects/{project_id}/files`
查询项目文件列表。

#### `GET /api/files/{file_id}`
查询文件详情。

#### `GET /api/files/{file_id}/download`
下载文件。

#### `DELETE /api/files/{file_id}`
删除文件。

#### `POST /api/files/{file_id}/parse`
触发文件解析任务。

---

### 4.5 用户端 Step API

#### Step 1：资料清单

##### `POST /api/projects/{project_id}/steps/step1/generate`
调用 `user_graphs.step1`，生成资料清单草稿。

##### `GET /api/projects/{project_id}/steps/step1/result`
获取 Step1 当前结果。

##### `GET /api/projects/{project_id}/steps/step1/history`
获取 Step1 历史版本。

##### `POST /api/projects/{project_id}/steps/step1/finalize`
确认 Step1 最终版。

---

#### Step 2：有效项目资料 / 核心内容

##### `POST /api/projects/{project_id}/steps/step2/generate`
调用 `user_graphs.step2`，生成核心内容草稿。

##### `GET /api/projects/{project_id}/steps/step2/result`
获取 Step2 当前结果。

##### `GET /api/projects/{project_id}/steps/step2/history`
获取 Step2 历史版本。

##### `POST /api/projects/{project_id}/steps/step2/finalize`
确认 Step2 最终版。

---

#### 后续步骤建议统一模式

例如：

- `POST /api/projects/{project_id}/steps/step3/generate`
- `POST /api/projects/{project_id}/steps/step4/generate`
- `POST /api/projects/{project_id}/steps/step5/generate`
- `POST /api/projects/{project_id}/steps/step14/generate`

统一采用：

- generate
- result
- history
- finalize

四类接口模式。

## 6. FastAPI 如何调用 agent

建议封装一层服务，例如：

- `AgentService.run_user_step(...)`
- `AgentService.run_admin_step(...)`

### 调用流程

1. 接收前端请求
2. 根据 `project_id` 查项目和文件
3. 组装 step 对应 state
4. 组装 runtime context
5. 调用对应 graph
6. 保存 step 输出与历史版本
7. 写入任务状态
8. 返回给前端 task_id 或结果

2. 项目创建
3. 文件上传
4. Step1 资料清单生成
5. Step2 核心内容生成
6. 历史版本保存
7. Word 导出
8. 管理端 Step1 配置生成

### MVP 暂缓项
- 全 14 步一次性做完
- 高级 PDF 排版
- 更复杂的知识库检索
- 全量统计报表
- 高级审批流

## 11. 最终结论

### 这个项目后端应该怎么做
- FastAPI 负责业务接口
- LangGraph 负责智能体执行
- PostgreSQL 负责数据持久化
- Redis 负责异步任务与缓存
- MinIO 负责文件存储

### 当前最应该先做什么
先把：
- `user_graphs.step1`
- `user_graphs.step2`
- `admin_graphs.step1`

做成 FastAPI API，再接数据库和任务系统。
